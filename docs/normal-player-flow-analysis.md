# 正常玩家登录后游玩流程分析

日期：2026-05-30

当前前提：

- 当前阶段仍以 Cocos-only 大厅为准，不回到 web-vue。
- 已有大厅能力：登录、资源加载、大厅 HUD、玩家资料只读弹窗、公告活动只读面板、图鉴只读面板、本地未开放占位弹窗。
- 已有只读接口：`POST /api/player/auth/dev-login`、`GET /api/player/me/lobby`、`GET /api/player/lobby/notices`、`GET /api/player/lobby/codex`。
- V1 规则边界：玩家等级上限 60，队伍最多 5 人，小怪战斗不扣体力，Boss 领奖扣体力，一阶副本不产资金池奖励，EX 只预埋不开放。
- 本分析只定义产品路径和后续开发顺序，不改变经济规则，不开放 EX V1，不新增经济写入口。

## 一、正常玩家登录后会做什么

玩家登录后不应该停在大厅里猜入口，标准流程应是：

```text
登录 -> 加载账号状态 -> 进入大厅 -> 明确下一目标 -> 进入主线/冒险 -> 编队确认 -> 战斗 -> 结算 -> 获得成长资源 -> 英雄/装备养成 -> 回到大厅继续下一目标
```

拆成玩家可感知动作：

1. 看自己的账号状态：等级、战力、体力、资源、公告。
2. 找到当前最重要目标：主线关卡、成长任务、日常任务或可挑战 Boss。
3. 进入战役/冒险：选择当前推荐关卡。
4. 确认队伍：查看队长、上阵英雄、战力差距。
5. 开始战斗：观看自动战斗或快速战斗。
6. 查看结算：经验、金币、材料、装备、任务进度。
7. 回到养成：升级英雄、强化装备、整理背包。
8. 再进入下一场战斗，形成循环。

## 二、多角色分析

### 产品角色

产品目标是建立第一条完整闭环，而不是同时打开所有大厅入口。

优先级建议：

1. 主线冒险闭环：大厅 `冒险/战役` -> 章节地图 -> 关卡详情 -> 队伍确认 -> 战斗 -> 结算 -> 返回大厅。
2. 队伍闭环：英雄列表 -> 默认队伍 -> 调整队长和站位 -> 保存队伍。
3. 成长闭环：战斗获得经验/金币/材料 -> 英雄升级或装备强化 -> 战力提升 -> 推进下一关。
4. 任务闭环：主线/日常任务读取 -> 完成状态 -> 奖励领取。
5. Boss/深渊闭环：等级达标后再开放，不应早于主线闭环。

当前下一阶段最适合做：

- 先做 `冒险/战役` 的只读章节与关卡选择壳。
- 再做 `队伍确认` 壳。
- 最后接入战斗与结算。结算涉及奖励发放，必须走后端事务，不能由前端本地发奖。

### 设计角色

设计核心是“目标、行动、反馈”。

- 大厅第一视觉焦点应是右下 `冒险` 主按钮，展示当前推荐目标，例如 `主线 1-1 暗影城门`。
- 左侧活动栏不应抢主线目标，活动只做次级入口。
- 底部导航要服务循环：英雄、背包、任务、锻造、商店、图鉴。
- 新玩家进入大厅时，可以用一条轻引导指向 `冒险`，避免强弹窗打断。
- 老玩家进入大厅时，优先展示可继续关卡、体力状态、任务可领奖、邮件/公告变化。

### 玩家角色

玩家登录后最关心三件事：

- 我现在能干什么？
- 干了有什么收益？
- 我为什么变强了？

新玩家期望：

- 5 秒内知道点哪里开始游戏。
- 不需要理解所有系统，先打一关。
- 第一场战斗后立刻看到等级、英雄经验或道具增长。

回流玩家期望：

- 看到体力是否满、任务是否可领、公告是否有新内容。
- 继续上次没打完的主线或挑战。
- 快速进入养成，不要被无关弹窗挡住。

### UI 角色

UI 需要把大厅从“展示页”推进到“操作中枢”。

建议新增或强化：

- 右下主按钮：显示当前推荐关卡、等级要求、体力/奖励提示。
- 主线推荐条：在大厅底部或右下按钮上方显示 `继续冒险`。
- 章节地图页：左侧章节列表，中间地图节点，右侧关卡信息。
- 关卡详情弹框：敌方等级、推荐战力、掉落预览、进入按钮。
- 队伍确认页：五个站位、队长标识、总战力、阵营羁绊提示。
- 战斗结算页：奖励列表、经验条变化、任务进度变化、下一关按钮。

UI 自适应要求不变：所有覆盖在背景/地图/战斗上的 UI 都必须基于安全区布局，不能写死屏幕坐标。

### 美术角色

美术方向应继续沿用暗黑哥特、黑金、深渊红的主调。

建议：

- 冒险地图使用暗色羊皮卷/深渊裂隙地图，不做明亮卡通地图。
- 关卡节点做成火盆、城门、裂缝、祭坛等场景物件。
- 战斗准备页用金属铭牌、暗红危险标记表达推荐战力差距。
- 结算页用红金光束和暗金边框，但避免像充值礼包弹窗。
- Boss/深渊入口需要更强压迫感，但应等主线闭环稳定后再做。

### 研发角色

研发要按“先只读、再状态写、最后经济结算”的顺序推进。

推荐技术阶段：

1. 大厅主线推荐状态：
   - 只读读取玩家当前主线进度、推荐关卡、等级要求。
   - 前端只展示，不改变玩家进度。
2. 冒险地图/关卡只读：
   - 读取章节和关卡配置。
   - 展示掉落预览，但不发奖。
3. 队伍只读与本地选择：
   - 读取英雄列表和已有队伍。
   - 本地调整站位，保存队伍属于玩家状态写入，需单独接口审查。
4. 战斗模拟：
   - 可以先做本地表现壳，但不能本地发奖。
   - 真正战斗结果和奖励必须后端结算。
5. 结算接口：
   - 后端校验关卡、队伍、体力、掉落、任务进度。
   - 后端事务写入经验、金币、材料、装备、任务进度。
   - 前端只展示后端返回结果。

### 审查角色

审查重点：

- 不允许前端决定奖励、掉落、经验、金币或材料。
- 不允许前端伪造关卡完成状态。
- 不允许把 EX V1 内容从图鉴、抽卡、奖励、活动中漏出来。
- 所有资源变更必须有后端事务、幂等 requestId、日志和异常回滚。
- 每个新接口必须标明是只读、玩家状态写入还是经济写入。
- 经济写入口必须单独审查，不和 UI 壳开发混在一个阶段。

### 验收角色

第一条完整游玩闭环的验收标准：

- 新玩家登录后 5 秒内能找到 `继续冒险`。
- 点击 `冒险` 可以进入章节地图。
- 点击当前关卡可以看到关卡详情和队伍确认。
- 队伍不足或等级不足时有明确提示。
- 战斗结束后结算展示清楚，奖励以服务端返回为准。
- 返回大厅后战力、等级、任务状态能刷新。
- 小屏、横屏、窄屏都不出现按钮越界、弹框误关闭或文字重叠。
- 未开放系统必须显示边界说明，不假装已有真实数据。

## 三、建议开发顺序

### Stage 3A：主线推荐入口

- 在大厅右下 `冒险` 按钮显示当前推荐主线。
- 接口优先只读：玩家当前主线进度、推荐关卡、等级要求。
- 未接后端前可用本地 placeholder，但 UI 结构按真实流程设计。

### Stage 3B：章节地图和关卡详情

- 新增章节地图页。
- 新增关卡详情弹框。
- 展示敌方等级、推荐战力、掉落预览、进入条件。
- 不发奖、不改变进度。

### Stage 3C：队伍确认

- 展示五个站位、队长、总战力、阵营羁绊。
- 没有队伍时给出本地引导。
- 保存队伍属于状态写入，需后端接口审查后再开放。

### Stage 3D：战斗表现壳

- 先做 Cocos 战斗场景、角色站位、基础技能表现和自动战斗节奏。
- 只用于表现验证，不产生奖励。

### Stage 3E：后端战斗结算

- 后端接管关卡完成、掉落、经验、任务进度和体力消耗。
- 前端展示结算结果。
- 这是第一个高风险经济写阶段，必须单独审查和测试。

## 四、当前结论

下一阶段不应该继续扩大厅的占位入口，而应该把 `冒险/战役` 做成第一条真实游玩主线。

最合理的下一个开发任务是：

```text
Stage 3A：大厅右下冒险入口 -> 主线推荐状态 -> 章节地图只读壳
```

这样可以让玩家从“能登录、能看大厅”进入“知道下一步、能开始玩”的阶段，同时仍保持经济规则和 EX V1 边界安全。
